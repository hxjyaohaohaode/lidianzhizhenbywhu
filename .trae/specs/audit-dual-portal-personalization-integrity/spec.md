# 双端个性化与数据通道完整性审计 Spec

## Why
当前系统已经具备企业端、普通用户端、会话上下文、用户画像、图表系统与多类分析接口，但仍缺少一次面向“数据通道、架构边界、页面分流、个性化服务差异”的系统级审计。现阶段需要详细检查系统内所有数据通道与页面链路，确认企业端与用户端不是“同一套界面换文案”，而是真正围绕不同角色目标提供不同的信息架构、内容来源、交互路径与个性化服务。
本次规格旨在建立一套可执行的审查与修正方案，系统性验证真实数据流向、角色隔离、个性化逻辑与页面差异化展示，避免企业端与用户端在内容、组件、入口或分析结果上出现混用、串用和弱个性化问题。

## What Changes
- 建立双端数据通道盘点与审计能力：
  - 盘点前端输入、前端 API、服务端入口、会话存储、画像存储、分析记录、图表载荷、外部数据源与导出链路。
  - 为每条数据通道标注所属角色、来源、用途、是否真实接口、是否存在模拟/降级逻辑、是否会影响双端个性化结果。
- 审查双端架构边界：
  - 明确企业端与普通用户端在 `App.tsx`、`api.ts`、`business-service.ts`、`platform-store.ts`、`session-store.ts`、`chart-data.ts` 中的职责边界。
  - 检查是否存在共享状态、共享组件或共享数据协议被错误复用，导致双端内容趋同或角色信息泄漏。
- 审查双端页面个性化：
  - 企业端重点检查数据采集、经营诊断、整改追踪、企业基础信息、经营类图表与动作入口。
  - 普通用户端重点检查画像采集、模式切换、研究判断、投资结论、证据链、对比能力与用户偏好映射。
  - 建立“页面级个性化矩阵”，逐页确认标题、模块、卡片、图表、按钮、文案、默认筛选和推荐逻辑是否按角色区分。
- 审查系统内容个性化链路：
  - 检查 `preferredRole`、`focusMode`、用户画像、企业基础信息、投资者基础信息、会话摘要、分析摘要、图表来源说明是否真正参与前端展示与后端结果生成。
  - 修复“有画像字段但未用于个性化展示”或“有角色字段但未驱动页面差异”的断链点。
- 增加双端隔离验收规则：
  - 建立针对角色错配、数据串流、个性化缺失、伪双端界面、模拟接口误展示等问题的验收门禁。

## Impact
- Affected specs: Dual Portal UX, Personalization, Session Context, User Profile, Data Governance, Chart System, API Boundary.
- Affected code: `src/web/App.tsx`、`src/web/api.ts`、`src/web/chart-data.ts`、`src/web/chart-system.tsx`、`src/shared/business.ts`、`src/server/app.ts`、`src/server/business-service.ts`、`src/server/platform-store.ts`、`src/server/session-store.ts`、`src/server/data/data-fetcher.ts`。

## ADDED Requirements
### Requirement: 双端数据通道审计清单
系统 SHALL 提供覆盖前端、服务端、存储层、图表层与外部数据源的双端数据通道审计清单，并明确每条链路服务于企业端、普通用户端或双端共享底座。

#### Scenario: 审查数据链路归属
- **WHEN** 研发审查任一接口、字段或存储记录
- **THEN** 能明确该链路的来源、目标页面、角色归属、是否真实接口、是否存在降级逻辑，以及是否参与个性化服务

### Requirement: 双端页面个性化矩阵
系统 SHALL 为企业端和普通用户端建立页面级个性化矩阵，逐页确认页面结构、内容优先级、图表类型、动作入口与默认文案的角色差异。

#### Scenario: 对比双端首页
- **WHEN** 审查企业端首页与普通用户端首页
- **THEN** 能明确两端首页在目标任务、默认模块、推荐卡片、主要图表、快捷动作和提示文案上的差异，且不能仅靠角色名称变化来区分

### Requirement: 个性化驱动链路完整性
系统 SHALL 确保用户画像、角色偏好、企业基础信息、投资者基础信息、会话上下文与分析模式真正参与双端内容生成与展示，而不是仅存储不生效。

#### Scenario: 画像字段参与展示
- **WHEN** 普通用户更新风险偏好、投资周期、关注企业或兴趣主题
- **THEN** 分析页默认模式、推荐摘要、图表排序、快捷问题和服务提示随之变化

#### Scenario: 企业信息参与展示
- **WHEN** 企业端补充企业基础信息、经营数据或经营关注点
- **THEN** 首页指标、分析摘要、整改入口、图表说明和建议重点按企业侧经营目标调整

### Requirement: 双端内容隔离校验
系统 SHALL 防止企业端内容、用户端内容、画像字段、按钮动作与服务提示发生错误串用或弱隔离。

#### Scenario: 角色切换后审查页面
- **WHEN** 同一用户在企业端和普通用户端之间切换
- **THEN** 各端页面仅展示本端应有的核心模块、文案和动作，不出现另一端专属内容被错误暴露的情况

### Requirement: 真实接口与模拟数据标识校验
系统 SHALL 明确标识真实接口、模拟数据、测试桩与降级数据，并确保其展示策略不会误导双端用户。

#### Scenario: 审查外部数据源
- **WHEN** 审查财报抓取、行业价格、宏观信号等外部数据链路
- **THEN** 能区分真实接入、模拟返回、待接入占位和失败降级，并在页面上采用一致的说明策略

## MODIFIED Requirements
### Requirement: 双端界面定义
现有“双端界面”要求从“存在企业端和普通用户端页面”升级为“存在两套围绕不同角色目标、内容结构、数据链路和服务策略的差异化界面”，并要求可通过页面矩阵和代码链路证明其差异。

### Requirement: 个性化服务定义
现有个性化服务要求从“根据角色展示部分差异内容”升级为“角色、画像、会话、关注点和模式切换共同驱动首页、分析页、设置页、图表区和快捷动作的动态差异”。

### Requirement: 数据治理定义
现有数据治理要求从“来源可追溯”升级为“来源可追溯 + 角色归属明确 + 个性化影响明确 + 真实/模拟状态明确”，确保数据通道不仅可追踪，而且可判断其是否正确服务于双端个性化。
