# 智能体深度融合与私有 API 治理 Spec

## Why
当前项目已经具备多智能体、记忆、双角色工作台和真实数据接入的基础能力，但这些能力仍分散在多个模块和历史变更中，缺少一次面向全项目的统一审查与收口。为了确保系统真正具备稳定的个性化服务能力，并满足“上游 API 与数据源凭证完全由平台内部托管、最终只需由运营侧填写必要 API 即可运行”的交付要求，需要新增一份聚焦集成治理的规格。

## What Changes
- 对全项目的智能体入口、任务流、个性化数据流、记忆复用链路、数据获取链路和配置边界进行一次系统化审查，并补齐缺口。
- 将智能体从“局部能力”升级为企业端、投资端、历史会话、记忆沉淀、证据生成和任务反馈中的默认核心执行机制。
- 建立服务端统一的上游模型/API/数据源配置治理，禁止将第三方 API 密钥、供应商地址或敏感凭证暴露给前端用户。
- 将个性化服务从单次会话增强为跨会话、跨任务、跨角色受控复用的持续服务能力。
- 建立“只需最后填写 API 即可运行”的交付标准，包括必填项、可选项、健康检查、连通性校验、降级策略和验收口径。
- **BREAKING**: 前端不得再承担第三方模型或外部数据源的配置职责，任何现存用户侧 API 输入入口都需要下沉或移除。

## Impact
- Affected specs: Agent 工作流、个性化服务、记忆体系、实时数据接入、系统配置治理、双角色业务流程
- Affected code: `src/server/env.ts`, `src/shared/config.ts`, `src/server/llm.ts`, `src/server/agent-service.ts`, `src/server/realtime-rag.ts`, `src/server/data/data-fetcher.ts`, `src/server/memory.ts`, `src/server/platform-store.ts`, `src/server/app.ts`, `src/web/api.ts`, `src/web/App.tsx`

## ADDED Requirements
### Requirement: 全项目智能体深度融合审查
系统 SHALL 对当前全部关键用户路径执行智能体接入审查，并确保核心分析、检索、记忆、表达和反馈链路都通过统一的智能体编排治理。

#### Scenario: 审查关键工作流的智能体覆盖率
- **WHEN** 平台审查企业端分析、投资端分析、历史会话复盘、记忆沉淀和证据生成流程
- **THEN** 系统能够明确每条链路是否经过统一编排器、使用了哪些智能体、存在何种旁路或缺失点
- **AND** 对未纳入治理的路径形成补位清单

#### Scenario: 关键分析默认走 Agent 编排
- **WHEN** 用户发起企业诊断、投资研究、追踪复盘或个性化建议请求
- **THEN** 系统默认通过任务编排、记忆召回、数据获取、证据审校和表达生成等智能体协作完成响应

### Requirement: 私有 API 与凭证服务端托管
系统 SHALL 将所有第三方模型 API、数据源凭证、供应商地址和认证材料保留在服务端配置层，并禁止普通用户在前端直接接触这些信息。

#### Scenario: 普通用户使用系统
- **WHEN** 企业用户或投资用户访问系统页面、发起任务或查看设置
- **THEN** 前端仅展示业务功能所需字段，不展示第三方 API Key、上游地址、账号密码或 Cookie 配置入口

#### Scenario: 平台维护外部连接
- **WHEN** 管理员或部署人员在服务端填写最终所需的模型 API 和数据源凭证
- **THEN** 系统能够在服务端完成统一校验、注册、健康检查与错误提示
- **AND** 前端无需新增任何供应商级配置即可正常运行

### Requirement: 个性化服务闭环
系统 SHALL 基于角色、历史会话、长期记忆、用户画像、行为反馈和任务上下文形成受控的个性化服务闭环。

#### Scenario: 老用户再次发起分析
- **WHEN** 已有历史记录的用户再次进入系统并发起新的分析任务
- **THEN** 系统能够自动召回相关记忆、偏好、关注指标和历史结论
- **AND** 输出内容在表达方式、证据排序、建议优先级和提醒内容上体现个性化差异

#### Scenario: 个性化与角色隔离并存
- **WHEN** 企业端与投资端共享同一平台底座
- **THEN** 系统能够复用用户级个性化资产
- **AND** 仍保证企业端和投资端的页面、任务、建议和证据展示不串端

### Requirement: 极简交付与可运行性验收
系统 SHALL 提供明确的最小部署与运行标准，确保项目在补齐必要的服务端 API 后即可稳定获取数据并完成主流程运行。

#### Scenario: 仅填写最终 API 后启动
- **WHEN** 部署人员按照规范填写服务端要求的最终模型 API 与必要数据源凭证
- **THEN** 系统能够通过启动校验和运行时健康检查确认关键依赖可用
- **AND** 主工作流在缺失可选凭证时执行受控降级而非整体失效

#### Scenario: 数据获取链路被验证
- **WHEN** 平台执行上线前验收
- **THEN** 系统能够验证真实数据获取、模型调用、记忆复用、个性化输出和前后端联调均通过

## MODIFIED Requirements
### Requirement: 配置与启动方式
系统 SHALL 从“用户或页面可能感知供应商配置”修改为“仅服务端持有并校验供应商配置”。最终交付时，部署人员只需要在服务端填写约定的 API 与必要凭证，普通用户不承担任何供应商接入责任。

### Requirement: Agent 工作流能力
系统 SHALL 从“部分功能使用 Agent 能力”修改为“所有关键分析与个性化主流程默认纳入统一 Agent 编排、追踪与治理”。

### Requirement: 个性化能力
系统 SHALL 从“以单次会话增强为主”修改为“以跨会话记忆、画像和反馈驱动的持续个性化服务为主”，并保留角色隔离与受控降级能力。

## REMOVED Requirements
### Requirement: 前端提供上游 API 或供应商配置入口
**Reason**: 该模式会增加用户使用复杂度，并带来安全与误配置风险，不符合平台内部托管的目标。
**Migration**: 将现存供应商配置项下沉至服务端环境配置、注册表或后台管理入口，前端仅保留业务级偏好设置与状态展示。
