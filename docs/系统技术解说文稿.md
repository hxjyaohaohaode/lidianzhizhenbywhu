# 第一部分：技术原理

## 1. 系统总体介绍

### 1.1 问题背景

锂电池行业作为新能源产业链的核心环节，近年来面临多重经营压力：
- **原材料价格波动剧烈**：碳酸锂价格从2022年高点60万元/吨跌至2024年不足10万元/吨，导致库存减值与成本倒挂
- **毛利率持续承压**：行业平均毛利率从25%下降至15%左右，部分企业甚至跌破10%
- **经营质量分化明显**：现金流、库存周转、资产负债率等指标呈现显著分化
- **投资决策复杂度提升**：传统财务分析难以快速识别风险信号与投资机会

### 1.2 系统定位

本系统定位为**面向锂电池企业毛利承压与经营质量变化的智能诊断平台**，核心能力包括：

1. **多维度经营诊断**：基于DQI（经营质量动态评价）和GMPS（毛利承压分析）双模型，实现企业健康度的量化评估
2. **智能体协同分析**：通过8个专业化智能体的协作，实现从数据采集到结论输出的全链路自动化
3. **实时行业情报整合**：基于RAG技术实时检索行业研报、政策信号、市场动态
4. **双Portal个性化服务**：面向企业运营分析人员和投资人员提供差异化服务

### 1.3 核心能力边界

| 能力维度 | 支持范围 | 不支持范围 |
|---------|---------|-----------|
| 企业诊断 | 毛利承压、经营质量、现金流安全 | 非锂电池行业企业 |
| 数据来源 | 企业财报、行业研报、宏观数据 | 实时交易数据、内部管理数据 |
| 分析深度 | 季度级别趋势分析 | 日级别实时监控 |
| 用户角色 | 企业运营分析、投资人员 | 监管机构、审计机构 |

## 3. 多智能体协作框架

### 3.1 智能体架构设计

本系统采用**多智能体协作架构**（Multi-Agent Collaboration Architecture），将复杂的企业诊断任务分解为8个专业化智能体的协同执行。

#### 3.1.1 智能体职责定义

| 智能体ID | 职责描述 | 核心能力 | 依赖关系 |
|---------|---------|---------|---------|
| `taskOrchestrator` | 任务编排 | 拆解目标、规划执行路径、协调并行任务 | 无 |
| `memoryManagement` | 记忆管理 | 召回历史记忆、维护上下文、保存新记忆 | 无 |
| `dataGathering` | 数据采集 | 自动化采集外部财报与宏观数据 | taskOrchestrator |
| `dataUnderstanding` | 数据理解 | 理解问题、评估数据完备性、识别缺失指标 | taskOrchestrator, memoryManagement |
| `mathAnalysis` | 数学分析 | 执行DQI/GMPS模型计算、生成量化结论 | taskOrchestrator, dataGathering |
| `industryRetrieval` | 行业检索 | 整合行业研报、政策信号、市场动态 | taskOrchestrator |
| `evidenceReview` | 证据审校 | 审校证据链、评估结论可信度 | dataUnderstanding, mathAnalysis, industryRetrieval |
| `expressionGeneration` | 表达生成 | 输出结构化结论与行动建议 | evidenceReview, memoryManagement |

### 4.3 模型路由机制

```typescript
export class ModelRouter {
  async execute(request: LlmExecutionRequest): Promise<RouterResult> {
    const providers = request.preferredProviders ?? defaultProviderOrder;
    const attempts: ProviderAttempt[] = [];
    const degradationTrace: DegradationEvent[] = [];

    for (const providerId of providers) {
      const adapter = this.adapters.get(providerId);
      if (!adapter?.isAvailable()) continue;
      
      try {
        const result = await adapter.complete(request);
        return { result, attempts, degradationTrace };
      } catch (error) {
        // 记录失败，继续尝试下一个模型
        degradationTrace.push({ agentId, reason: "provider_failed", ... });
      }
    }
    
    throw new ModelExecutionError({ message: "所有模型均不可用", attempts, degradationTrace });
  }
}
```

## 6. 数学模型详解

### 6.1 DQI经营质量动态评价模型

**【原创工作】** DQI（Dynamic Quality Index）模型是本系统的核心原创贡献，用于量化评估企业经营质量的动态变化。

#### 6.1.1 模型原理

DQI模型基于杜邦分析体系，将经营质量分解为三个核心维度：

$$DQI = w_1 \cdot R_{ROE} + w_2 \cdot R_{Growth} + w_3 \cdot R_{OCF}$$

其中：
- $R_{ROE}$ = 当期ROE / 基期ROE（盈利能力比率）
- $R_{Growth}$ = 当期营收增长率 / 基期营收增长率（成长能力比率）
- $R_{OCF}$ = 当期OCF比率 / 基期OCF比率（现金流质量比率）
- $w_1, w_2, w_3$ 为权重系数，默认值为 $w_1=0.4, w_2=0.3, w_3=0.3$

#### 6.1.2 ROE计算

$$ROE = \frac{净利润}{平均净资产} \times 100\%$$

$$平均净资产 = \frac{期初净资产 + 期末净资产}{2}$$

#### 6.1.3 驱动因素识别

采用argmax方法识别主要驱动因素：

```typescript
function identifyDriver(roeRatio, growthRatio, ocfRatio): DQIDriver {
  const contributions = [
    { name: "盈利能力", value: Math.abs(roeRatio - 1) },
    { name: "成长能力", value: Math.abs(growthRatio - 1) },
    { name: "现金流质量", value: Math.abs(ocfRatio - 1) },
  ];
  const maxDeviation = Math.max(...contributions.map(c => c.value));
  if (maxDeviation < 0.05) return "无明显驱动";
  return contributions.reduce((max, current) => 
    current.value > max.value ? current : max
  ).name;
}
```

#### 6.1.4 状态判定

| DQI值范围 | 状态 | 含义 |
|----------|------|------|
| DQI > 1.05 | 改善 | 经营质量较基期显著提升 |
| 0.95 ≤ DQI ≤ 1.05 | 稳定 | 经营质量维持基期水平 |
| DQI < 0.95 | 恶化 | 经营质量较基期下降 |

### 6.2 GMPS毛利承压分析模型

**【原创工作】** GMPS（Gross Margin Pressure Score）模型是本系统的另一核心原创贡献，专门针对锂电池企业的毛利承压问题设计。

#### 6.2.1 五层十维特征体系

```
A层：毛利率结果（权重0.25）
├── gpmYoy: 毛利率同比变化
└── revCostGap: 营收成本增速差

B层：材料成本冲击（权重0.22）
├── liPriceYoy: 碳酸锂价格同比变化
└── unitCostYoy: 单位成本同比变化

C层：产销负荷（权重0.31）
├── invYoy: 库存同比变化
├── saleProdRatio: 产销率
└── mfgCostRatio: 制造费用占比

D层：外部风险（权重0.07）
└── indVol: 行业指数波动率

E层：现金流安全（权重0.15）
├── cfoRatio: 经营现金流比率
└── lev: 资产负债率
```

#### 6.2.2 特征打分函数

**越大越危险的指标**（如成本增速差、库存增长）：

$$Score_{increase}(x) = \begin{cases} 20 & x \leq lowRisk \\ 80 & x \geq highRisk \\ 20 + \frac{x - lowRisk}{highRisk - lowRisk} \times 60 & otherwise \end{cases}$$

**越小越危险的指标**（如产销率、现金流比率）：

$$Score_{decrease}(x) = \begin{cases} 80 & x \leq lowRisk \\ 20 & x \geq highRisk \\ 80 - \frac{x - lowRisk}{highRisk - lowRisk} \times 60 & otherwise \end{cases}$$

#### 6.2.3 GMPS综合得分

$$GMPS = \sum_{i=1}^{10} w_i \cdot Score_i$$

其中权重配置为：

```typescript
const weights = {
  gpmYoy: 0.14, revCostGap: 0.11,
  liPriceYoy: 0.10, unitCostYoy: 0.12,
  invYoy: 0.09, saleProdRatio: 0.10, mfgCostRatio: 0.12,
  indVol: 0.07,
  cfoRatio: 0.08, lev: 0.07,
};
```

#### 6.2.4 Logistic风险预测

基于GMPS得分和各维度得分，预测下季度风险概率：

$$P = \sigma(z) = \frac{1}{1 + e^{-z}}$$

$$z = \beta_0 + \beta_1 \cdot GMPS + \beta_2 \cdot S_A + \beta_3 \cdot S_B + \beta_4 \cdot S_C + \beta_5 \cdot S_D + \beta_6 \cdot S_E$$

默认参数：

```typescript
const beta = {
  beta0: -2.5,
  beta1: 0.05,  // GMPS系数
  beta2: 0.02,  // A层系数
  beta3: 0.03,  // B层系数
  beta4: 0.02,  // C层系数
  beta5: 0.01,  // D层系数
  beta6: 0.02,  // E层系数
};
```

#### 6.2.5 等级划分

| GMPS得分范围 | 承压等级 | 风险概率范围 | 风险等级 |
|-------------|---------|-------------|---------|
| < 40 | 低压 | < 0.33 | 低风险 |
| 40 - 70 | 中压 | 0.33 - 0.66 | 中风险 |
| ≥ 70 | 高压 | ≥ 0.66 | 高风险 |

### 7.3 记忆召回策略

**【原创工作】** 记忆召回策略结合用户画像、会话上下文和历史结论：

```typescript
async recallMemories(userId: string, query: string): Promise<MemoryEntry[]> {
  // 1. 基于用户ID召回所有记忆
  const allMemories = await this.store.getByUserId(userId);
  
  // 2. 基于查询关键词过滤
  const queryTerms = this.extractKeyTerms(query);
  const relevantMemories = allMemories.filter(m => 
    this.hasOverlap(m.tags, queryTerms) || 
    this.hasOverlap(this.extractKeyTerms(m.summary), queryTerms)
  );
  
  // 3. 按时间排序，返回最近的N条
  return relevantMemories
    .sort((a, b) => new Date(b.createdAt).getTime() - new Date(a.createdAt).getTime())
    .slice(0, 5);
}
```

```typescript
function normalizeHigherBetter(value: number, min: number, max: number) {
  if (min === max) return 100;
  return round(clamp(((value - min) / (max - min)) * 100, 0, 100));
}

function normalizeLowerBetter(value: number, min: number, max: number) {
```
