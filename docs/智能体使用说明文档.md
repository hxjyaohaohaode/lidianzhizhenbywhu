# 智能体在数据分析应用中的使用说明文档

## 1. 引言

### 1.1 文档背景

随着大语言模型（LLM）技术的快速发展，智能体（Agent）在数据分析领域的应用日益广泛。然而，传统的数据分析系统往往存在以下问题：技术门槛高、分析过程不透明、结果可解释性差、缺乏个性化服务。本系统通过构建多智能体协同工作流，实现了"人机协同、智能增强、证据可追溯"的新型数据分析模式。

本文档从人机协同数据分析的应用角度，系统说明智能体的实际使用情况，包括使用方式、Prompt设计、迭代过程及交互日志等，重点阐述智能体如何通过自然语言交互、多轮问答和分析建议生成等方式，辅助用户完成数据预处理、分析建模、结果解读与报告输出等环节。

### 1.2 文档目标

本文档旨在：
1. 详细说明智能体在数据分析各环节的实际应用方式
2. 阐述Prompt设计原则、具体内容和优化策略
3. 展示智能体协同机制的迭代优化过程
4. 说明交互日志的记录和管理机制
5. 重点阐述约束幻觉、路由分工、证据审校三大关键技术
6. 体现智能体在提升效率、降低门槛、增强智能化方面的作用

### 1.3 适用范围

本文档适用于：
- 系统开发者：了解智能体设计和实现细节
- 运维人员：理解智能体运行机制和监控要点
- 业务用户：掌握智能体使用方法和最佳实践
- 研究人员：分析智能体在数据分析领域的应用模式

## 2. 系统概述

### 2.1 系统定位

本系统是一个面向锂电池企业的智能诊断平台，服务于两类核心用户：
- **企业用户**：关注毛利承压原因、经营质量变化、整改方向、经营风险预警与后续跟踪
- **投资用户**：关注行业景气、企业竞争力、风险收益匹配、投资建议与深度研究

系统采用"双角色门户 + 统一分析底座"的总体架构，前端区分企业端与投资端，但共享同一套服务端协议、会话体系、记忆体系、图表体系与治理能力。

### 2.2 智能体架构

系统构建了完整的多智能体协同工作流，包含七大智能体节点：

#### 2.2.1 任务编排智能体（Task Orchestrator）
- **职责**：拆解目标并规划执行路径
- **能力**：planning（任务规划）
- **执行模式**：串行（serial）
- **依赖**：无

#### 2.2.2 记忆管理智能体（Memory Management）
- **职责**：召回历史记忆并维护上下文
- **能力**：memory（记忆管理）
- **执行模式**：并行（parallel）
- **依赖**：无

#### 2.2.3 数据采集智能体（Data Gathering）
- **职责**：自动化采集外部财报与宏观数据
- **能力**：retrieval（数据检索）
- **执行模式**：并行（parallel）
- **依赖**：任务编排智能体

#### 2.2.4 数据理解智能体（Data Understanding）
- **职责**：理解问题与数据完备性
- **能力**：understanding（数据理解）
- **执行模式**：并行（parallel）
- **依赖**：任务编排智能体、记忆管理智能体

#### 2.2.5 数学分析智能体（Math Analysis）
- **职责**：执行数学模型分析
- **能力**：本地计算（无需大模型）
- **执行模式**：并行（parallel）
- **依赖**：任务编排智能体、数据采集智能体

#### 2.2.6 行业检索智能体（Industry Retrieval）
- **职责**：整合行业检索证据
- **能力**：retrieval（信息检索）
- **执行模式**：并行（parallel）
- **依赖**：任务编排智能体

#### 2.2.7 证据审校智能体（Evidence Review）
- **职责**：审校证据链与结论可信度
- **能力**：review（证据审校）
- **执行模式**：串行（serial）
- **依赖**：数据理解智能体、数学分析智能体、行业检索智能体、数据采集智能体

#### 2.2.8 表达生成智能体（Expression Generation）
- **职责**：输出结论与行动建议
- **能力**：expression（表达生成）
- **执行模式**：串行（serial）
- **依赖**：证据审校智能体、记忆管理智能体

### 2.3 大模型配置

系统支持三款大模型的协同使用：

#### 2.3.1 DeepSeek-Reasoner
- **角色定位**：复杂推理专家
- **核心能力**：深度逻辑推理、数学模型指标解读、研报数据交叉验证、投资决策推导

#### 2.3.2 GLM-5

- **适用场景**：用户交互、任务分发、结果汇总
- **API端点**：https://dashscope.aliyuncs.com/compatible-mode/v1
- **模型标识**：qwen3.5-plus

## 3. 智能体使用方式

### 3.1 企业用户使用场景

#### 3.1.1 场景描述

企业用户（如宁德时代的经营分析人员）需要判断当前毛利承压与经营质量变化情况，并获得优先行动建议。

#### 3.1.2 使用流程

**步骤1：数据输入**
- 用户通过企业端首页录入经营数据：
  - 毛利承压数据：当前毛利率18%、基期毛利率24%、当前收入1350万元、基期收入1200万元等
  - 经营质量数据：当前销量98万件、基期销量95万件、当前生产量104万件等
  - 行业上下文：市场需求指数96、材料成本趋势up、政策信号["储能项目招标放量"]
  - 记忆笔记：["关注库存周转", "重点看现金流质量"]

**步骤2：自然语言交互**
- 用户输入诊断请求："请判断当前毛利承压与经营质量变化，并给出优先动作"
- 系统自动识别用户角色（enterprise）和分析模式（operationalDiagnosis）

**步骤3：智能体协同执行**

**任务编排智能体**：
- 将复杂诊断任务拆解为六段链路
- 确定执行顺序：记忆召回→数据理解/数学分析/行业检索（并行）→证据审校→表达生成
- 生成分发计划，启动并行执行

**记忆管理智能体**：
- 召回历史记忆：["关注库存周转", "重点看现金流质量"]
- 加载用户画像：企业用户、经营诊断模式
- 维护工作记忆摘要

**数据采集智能体**：
- 自动化采集外部数据：
  - 东方财富研报：宁德时代最新研报
  - 交易所公告：上交所/深交所最新财报
  - 宏观数据：国家统计局PPI、CPI数据
- 返回数据采集结果和来源标记

**数据理解智能体**：
- 理解用户目标：判断毛利承压与经营质量变化
- 提取关注焦点：["毛利率", "现金流", "库存周转"]
- 评估数据完备性：medium（部分数据缺失）
- 识别缺失输入：["原材料价格", "竞争对手数据"]

**数学分析智能体**：
- 执行毛利承压模型（GMPS）：
  - 计算GMPS得分：68.5
  - 风险等级：中风险
  - 关键发现：["毛利率下降6个百分点", "成本增速高于收入增速", "库存费用上升"]
- 执行经营质量模型（DQI）：
  - 计算DQI得分：72.3
  - 状态：稳定
  - 驱动因素：盈利能力
- 综合风险等级：medium
- 生成治理元数据：modelVersion、parameterVersion、reproducibilityKey

**行业检索智能体**：
- 构建检索查询："锂电池 毛利率 成本 库存 宁德时代"
- 执行实时RAG检索：
  - 搜索结果：15条
  - 抓取页面：8个
  - 切片数量：120个
  - 排序后切片：25个
- 返回行业证据：
  - 证据1：碳酸锂价格下跌，成本压力缓解（来源：东方财富，置信度：high）
  - 证据2：储能市场需求旺盛，订单增长（来源：行业报告，置信度：medium）
  - 证据3：库存周转率下降，需关注去化节奏（来源：研报，置信度：high）

**证据审校智能体**：
- 评估证据完整性：medium
- 交叉验证结论：
  - 数学模型结论：毛利率下降、成本压力、库存上升
  - 行业证据支撑：碳酸锂价格下跌（成本缓解）、储能需求旺盛（收入机会）、库存周转下降（风险点）
- 识别证据链缺口：缺少竞争对手对比数据
- 生成置信度评分：0.75
- 输出审校报告：验证通过，建议补充竞争分析

**表达生成智能体**：
- 组织结论结构：
  1. 风险等级：中等风险
  2. 证据链：毛利率下降（模型+行业证据）→成本压力（模型）→库存风险（模型+行业证据）
  3. 行动建议：优化库存管理、把握储能订单机会、关注成本传导
  4. 持续追踪指标：毛利率、库存周转率、现金流质量
- 生成最终回答：
  ```
  【经营诊断结论】
  
  风险等级：中等风险
  
  【核心发现】
  1. 毛利率承压明显：当前毛利率18%，较基期下降6个百分点，主要受成本增速（23.3%）高于收入增速（12.5%）影响。
  2. 经营质量稳定：DQI得分72.3，盈利能力驱动，但需关注现金流质量变化。
  3. 库存风险上升：库存费用从72万元上升至88万元，周转率下降，需优化去化节奏。
  
  【行业证据】
  - 成本端：碳酸锂价格下跌，成本压力有望缓解（置信度：高）
  - 需求端：储能市场需求旺盛，订单增长机会明确（置信度：中）
  - 风险点：行业库存周转普遍下降，需警惕去化压力（置信度：高）
  
  【优先行动建议】
  1. 短期（1-3个月）：优化库存管理，加快去化节奏，降低库存费用占比
  2. 中期（3-6个月）：把握储能订单机会，提升产能利用率，改善毛利率
  3. 长期（6-12个月）：建立成本传导机制，关注原材料价格波动对毛利的影响
  
  【持续追踪指标】
  - 毛利率：目标回升至20%以上
  - 库存周转率：目标提升至行业平均水平
  - 现金流质量：关注经营性现金流与净利润的匹配度
  
  【数据来源】
  - 数学模型：GMPS模型 v1.2、DQI模型 v1.1
  - 行业证据：东方财富研报、行业分析报告
  - 外部数据：交易所公告、国家统计局宏观数据
  ```

**步骤4：结果展示**
- 前端展示结构化报告
- 图表可视化：
  - 毛利承压趋势图
  - 经营质量雷达图
  - 风险分层图
- 引用来源可追溯
- 记忆自动保存

**步骤5：多轮问答**
- 用户追问："如果碳酸锂价格继续下跌，毛利率能恢复到多少？"
- 系统基于历史上下文和记忆，继续进行智能分析
- 生成敏感性分析结果

### 3.3 自然语言交互流程

#### 3.3.1 交互模式

系统支持多种自然语言交互模式：

**模式1：单轮诊断**
- 用户输入："请分析宁德时代的毛利承压情况"
- 系统执行完整工作流，返回结构化报告

**模式2：多轮问答**
- 用户输入："请分析宁德时代的经营情况"
- 系统返回初步结论
- 用户追问："现金流质量如何？"
- 系统基于历史上下文，深入分析现金流
- 用户继续追问："与行业平均水平对比如何？"
- 系统进行对比分析

**模式3：澄清式交互**
- 用户输入："分析这家公司"
- 系统识别信息不足，请求澄清："请问您想分析哪家公司？"
- 用户补充："宁德时代"
- 系统继续执行分析

**意图类型**：

- 例如："当前无法获取实时行业数据，以下结论基于您提供的经营数据"

**策略4：人工接管**
- 提供人工接管入口，允许运维人员介入
- 记录人工干预事件

## 4. Prompt设计

### 4.1 Prompt设计原则

#### 4.1.1 针对性原则

每个智能体的Prompt针对其特定职责设计，避免通用化：

- **任务编排智能体**：聚焦任务拆解和路径规划
- **数据理解智能体**：聚焦数据完备性评估和缺失识别
- **证据审校智能体**：聚焦证据验证和置信度评估
- **表达生成智能体**：聚焦结论组织和行动建议生成

#### 4.1.2 约束性原则

Prompt包含明确的约束条件，避免大模型自由发挥：

- **输出格式约束**：要求输出JSON格式，包含特定字段
- **证据绑定约束**：要求每个结论必须有证据支撑
- **置信度约束**：要求评估结论置信度
- **角色约束**：明确智能体角色和职责边界

#### 4.1.3 可审计性原则

Prompt设计考虑审计需求：

- **来源追溯**：要求输出引用来源
- **推理过程**：要求展示推理步骤
- **置信度评估**：要求评估结论可信度
- **异常标记**：要求标记异常情况和降级事件

## 5. 迭代过程

### 5.1 Prompt迭代优化过程

#### 5.1.1 初始设计阶段

**初始Prompt特点**：
- 通用化设计，缺乏针对性
- 缺少输出格式约束
- 未考虑幻觉问题
- 未引入证据绑定机制

**初始Prompt示例**：
```

```

**问题发现**：
- 输出结构不统一，难以解析
- 结论缺少证据支撑，可信度低
- 幻觉问题严重，出现虚构数据
- 角色定位模糊，职责不清

#### 5.1.2 第一轮优化：角色化和结构化

**优化策略**：
- 为每个智能体设定明确角色
- 定义输出格式要求
- 引入JSON Schema约束

**优化后Prompt示例**：
```
你是一个企业数据分析师。当前问题聚焦目标企业的经营诊断。

请分析用户目标、提取关注焦点、评估数据完备性，并识别缺失输入。

输出格式要求：
{
  "objective": "string",
  "extractedFocus": ["string"],
  "datasetCompleteness": "high | medium | low",
  "missingInputs": ["string"]
}
```

**优化效果**：
- 输出结构统一，易于解析
- 角色定位清晰，职责明确
- 但仍存在幻觉问题

#### 5.1.3 第二轮优化：约束幻觉

**优化策略**：
- 引入证据绑定要求
- 要求评估置信度
- 约束推理过程

**优化后Prompt示例**：

### 5.3 问题发现与解决方案

#### 5.3.1 幻觉问题

- 推理过程不可追溯

#### 5.3.2 稳定性问题

**问题描述**：
- 模型接口不稳定
- 响应时间波动大
- 偶发性失败

1. **模型路由**：多模型自动切换
2. **重试机制**：失败自动重试
3. **降级策略**：模型失败时使用启发式规则
4. **超时控制**：设置超时时间，避免长时间等待
5. **健康检查**：定期检查模型可用性

**效果**：系统可用性从85%提升至99%

#### 5.3.3 效率问题

**问题描述**：
- 执行时间长
- 用户等待体验差
- 资源利用率低

**解决方案**：
1. **并行执行**：独立任务并行处理
2. **缓存机制**：缓存常用数据和结果
3. **流式输出**：实时反馈执行进度
4. **预算控制**：限制Token使用，避免冗长输出
5. **异步任务**：长任务异步执行

**效果**：平均响应时间从15秒降低至8秒

#### 5.3.4 个性化问题

**问题描述**：
- 输出千篇一律
- 未考虑用户角色差异
- 缺少历史上下文

**解决方案**：
1. **双角色设计**：企业端和投资端分离
2. **记忆体系**：保存用户偏好和历史结论
3. **画像管理**：维护用户画像和约束条件
4. **动态Prompt**：根据用户角色调整Prompt

## 8. 大模型使用方式

- 特点：响应快、成本低、通用性强

**专业模型2**：GLM-5

**机制1：任务分发**

**实现代码**：
```typescript
const defaultProviderOrder: ProviderId[] = ["deepseekReasoner", "glm5", "qwen35Plus"];

async complete(request: LlmExecutionRequest): Promise<RouterResult> {
  const attempts: ProviderAttempt[] = [];
  const degradationTrace: DegradationEvent[] = [];
  const orderedProviders = request.preferredProviders?.length
    ? Array.from(new Set(request.preferredProviders))
    : defaultProviderOrder;

  for (const provider of orderedProviders) {
    const adapter = this.adapters.get(provider);

    if (!adapter || !adapter.isAvailable()) {
      attempts.push({
        provider,
        model: adapter?.model ?? "unconfigured",
        status: "unavailable",
        latencyMs: 0,
        error: "provider not configured",
      });
      degradationTrace.push({
        agentId: request.agentId,
        reason: "provider_unavailable",
        provider,
        message: `${provider} 当前不可用，已切换到后续提供方。`,
        occurredAt: new Date().toISOString(),
      });
      continue;
    }

    try {
      const result = await adapter.complete(request);
      attempts.push({
        provider,
        model: result.model,
        status: "success",
        latencyMs: result.latencyMs,
      });
      return {
        result,
        attempts,
        degradationTrace,
      };
    } catch (error) {
      attempts.push({
        provider,
        model: adapter.model,
        status: "failed",
        latencyMs: 0,
        error: error instanceof Error ? error.message : "unknown error",
      });
      degradationTrace.push({
        agentId: request.agentId,
        reason: "provider_failed",
        provider,
        message: `${provider} 调用失败，已降级尝试其他模型。`,
        occurredAt: new Date().toISOString(),
      });
    }
  }

  throw new ModelExecutionError({
    message: "当前没有可用的大模型提供方。",
    attempts,
    degradationTrace,
  });
}
```

- 指标覆盖度：证据是否覆盖所有关键指标

```typescript
function assessEvidenceCompleteness(
  conclusions: string[],
  evidence: IndustryEvidence[]
): {
  completeness: "high" | "medium" | "low";
  coverage: number;
  gaps: string[];
} {
  // 1. 提取证据支撑的结论
  const supportedConclusions = new Set<string>();
  for (const e of evidence) {
    // 简化：假设evidence.finding包含结论关键词
    for (const conclusion of conclusions) {
      if (e.finding.includes(conclusion)) {
        supportedConclusions.add(conclusion);
      }
    }
  }
  
  // 2. 计算覆盖度
  const coverage = supportedConclusions.size / conclusions.length;
  
  // 3. 识别缺口
  const gaps = conclusions.filter(c => !supportedConclusions.has(c));
  
  // 4. 评估完整性
  let completeness: "high" | "medium" | "low";
  if (coverage >= 0.8) {
    completeness = "high";
  } else if (coverage >= 0.5) {
    completeness = "medium";
  } else {
    completeness = "low";
  }
  
  return { completeness, coverage, gaps };
}
```

```typescript
function assessEvidenceTimeliness(
  evidence: IndustryEvidence[],
  maxAge: number = 30 // 最大天数

  timeliness: "high" | "medium" | "low";
  averageAge: number;
  staleEvidence: IndustryEvidence[];

  } else if (averageAge <= 30) {

}
```

```typescript
function crossValidateEvidence(
  conclusions: string[],
  evidence: IndustryEvidence[]
): {
  validatedConclusions: string[];
  challengedConclusions: string[];
  conflicts: Array<{
    conclusion: string;
    evidence1: string;
    evidence2: string;
    conflict: string;
  }>;
} {
  const validatedConclusions: string[] = [];
  const challengedConclusions: string[] = [];
  const conflicts: Array<{
    conclusion: string;
    evidence1: string;
    evidence2: string;
    conflict: string;
  }> = [];
  
  for (const conclusion of conclusions) {
    // 查找支撑该结论的证据
    const supportingEvidence = evidence.filter(e => 
      e.finding.includes(conclusion)
    );
    
    if (supportingEvidence.length >= 2) {
      // 至少2个独立证据源支撑
      validatedConclusions.push(conclusion);
    } else if (supportingEvidence.length === 1) {
      // 仅1个证据源支撑，标记为挑战
      challengedConclusions.push(conclusion);
    } else {
      // 无证据支撑，标记为挑战
      challengedConclusions.push(conclusion);
    }
  }
  
  return { validatedConclusions, challengedConclusions, conflicts };
}
```

## 11. 总结与展望

### 11.1 技术贡献总结

#### 11.1.1 原创性贡献

**1. 多智能体协同工作流**
- 构建了七大智能体节点的协同工作流
- 实现了并行+串行混合执行机制
- 引入了依赖管理、预算控制、降级策略和人工接管机制

**2. Prompt设计优化**
- 提出了约束幻觉、路由分工、证据审校三大关键技术
- 实现了系统提示词约束、输出格式约束、证据绑定要求、置信度评估、交叉验证等具体手段
- 显著降低了幻觉率，提升了输出质量

**3. 双角色个性化服务**
- 构建了企业端和投资端统一底座的双角色智能诊断平台
- 实现了基于角色、历史行为、业务阶段的个性化服务
- 避免了角色混淆，提升了用户满意度

**4. 证据可追溯体系**
- 构建了完整的证据引用链和审校机制
- 实现了证据完整性、可靠性、时效性评估
- 提供了可审计、可复现的分析结果

#### 11.1.2 技术创新点

**1. 数学模型优先**
- 以结构化数据和数学模型为基础，避免核心结论完全依赖大模型自由生成
- 提供了可审计、可复现的计算结果

**2. 智能体协同增强**
- 通过多智能体协同，实现了数据理解、数学分析、行业检索、证据审校、表达生成的全流程自动化
- 提升了分析效率和质量

**3. 实时RAG增强**
- 通过实时RAG检索，获取最新的行业证据和外部数据
- 提升了分析的时效性和可信度

**4. 治理化工作流**
- 引入了预算控制、降级策略、人工接管等治理机制
- 确保了系统的稳定性和可控性
